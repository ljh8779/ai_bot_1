## 한 줄로 말하면
- 이 프로젝트는 **회사 문서를 읽어 두었다가**, 사람이 질문하면 **관련 문서를 찾아서 답해주는 로봇**입니다.

## 누가 일하나요?
1. 사용자: 질문을 하는 사람
2. 웹 화면(React): 질문/업로드 버튼이 있는 화면
3. API 서버(FastAPI): 전체 일을 지휘하는 반장
4. 문서 창고(PostgreSQL + pgvector): 문서 조각을 보관하는 창고
5. AI 엔진(Google Gemini 또는 Ollama): 글을 이해하고 답을 만드는 두뇌

## 전체 워크플로우
1. 서버가 시작되면 건강검진을 합니다.
2. 문서를 넣습니다. (텍스트/파일/폴더)
3. 문서를 작은 조각으로 나눕니다.
4. 각 조각을 숫자 벡터로 바꿔 창고에 저장합니다.
5. 사용자가 질문하면, 비슷한 조각을 창고에서 찾습니다.
6. 찾은 조각을 AI에게 보여주고 최종 답변을 받습니다.

## 1) 문서 넣기(Ingest) 흐름
1. 사용자가 문서를 넣습니다.
2. `POST /documents/text`: 직접 텍스트 입력
3. `POST /documents/file`: 파일 업로드(.txt, .md, .pdf, .pptx, 이미지)
4. `POST /documents/bulk`: 폴더/ZIP 한꺼번에 처리
5. 서버가 텍스트를 뽑아냅니다.
6. PDF는 일반 텍스트 추출 후, 글자가 너무 적으면 OCR도 시도합니다.
7. 이미지는 OCR로 텍스트를 읽습니다.
8. 텍스트를 작은 조각으로 자릅니다.
9. 기본값: `chunk_size=800`, `chunk_overlap=120`
10. 같은 문서인지 확인합니다.
11. 내용 해시(`content_hash`)가 같으면 중복 저장을 피합니다.
12. 조각마다 임베딩(숫자 벡터)을 만들고 DB에 저장합니다.

## 2) 질문-답변(RAG Chat) 흐름
1. 사용자가 질문합니다. (`POST /chat`)
2. 최근 대화가 있으면, 후속 질문인지 판단해 질문을 더 명확하게 바꿉니다.
3. 질문을 임베딩으로 바꿉니다.
4. 벡터 검색으로 비슷한 문서 조각 후보를 찾습니다.
5. 후보 개수 기본값: `max_context_chunks(5) x multiplier(4) = 20`
6. 권한 필터를 적용합니다.
7. `allowed_departments`, `allowed_roles`와 사용자 정보를 비교합니다.
8. 최종 상위 조각(기본 5개)을 AI에게 컨텍스트로 전달합니다.
9. 점수가 너무 낮거나 결과가 없으면 일반 답변 모드로 전환할 수 있습니다.
10. 기본 기준: `GENERAL_FALLBACK_MIN_SCORE=0.70`
11. 최종 답변과 출처 조각을 사용자에게 보여줍니다.

## 3) 권한(ACL)
- 문서에 "이 부서만", "이 역할만" 스티커를 붙일 수 있습니다.
- 질문한 사람의 부서/역할이 맞아야 문서를 답변에 사용합니다.
- 보이면 안 되는 문서는 자동 제외됩니다.

## 핵심 파일 지도
- `app/main.py`: API 입구와 라우팅
- `app/services/file_extract.py`: 파일에서 글자 읽기
- `app/services/rag.py`: 문서 저장 + 검색 + 답변 핵심
- `app/services/llm.py`: Google/Ollama 연결
- `app/models.py`: 문서/문서조각 DB 구조
- `app/web/src/App.jsx`: 실제 웹 UI 동작
